### Phase 1.

### **Task 1: Project Initialization & Modern Scaffolding**
We start by generating the raw Next.js project, but we will immediately strip away the standard boilerplate and configure it to use the fastest modern package manager (we will use `bun` or `pnpm` to show we are up to date with 2026 performance standards).

*   **Sub-task 1.1: Package Manager & Environment Setup.** Define the explicit Node.js and Package Manager versions. We will create a `.nvmrc` and `.npmrc` (or equivalent) to enforce that anyone (or any CI/CD pipeline) cloning this repo uses the exact same environment. This prevents the classic "it works on my machine" bug.
*   **Sub-task 1.2: Execute `create-next-app`.** We will run the initialization script with strict flags: enforcing TypeScript, Tailwind CSS, App Router (`/src` directory), and custom absolute path aliases (`@/*`).
*   **Sub-task 1.3: The "Scorched Earth" Cleanup.** Standard Next.js templates come with generic CSS and layout boilerplate. We will delete all default CSS, clear the `page.tsx`, and strip the `layout.tsx` down to its bare `<html>` and `<body>` tags. We start from an absolute blank slate.

### **Task 2: Next.js Static Export Architecture (GitHub Pages Prep)**
GitHub Pages cannot run a Node.js server; it only serves static assets. Next.js natively uses server-side rendering (SSR), so we must fundamentally alter its build behavior to output a purely static site.

*   **Sub-task 2.1: Configure `next.config.mjs`.** We will set the `output: 'export'` property. This instructs Next.js to compile all Server Components into raw HTML/CSS and JavaScript bundles during the build step.
*   **Sub-task 2.2: Image Optimization Bypass.** Next.js's native `<Image>` component relies on a Node server to compress images on the fly. Since we are doing a static export, this will crash the build. We will configure the `images: { unoptimized: true }` property in the config, ensuring we can still use the `<Image>` tag for layout stability (preventing Cumulative Layout Shift) without breaking the static build.
*   **Sub-task 2.3: Repository Path Mapping (Optional but critical).** If your GitHub Pages will be hosted at `username.github.io/portfolio` (a project page) rather than `username.github.io` (a user page), we must configure the `basePath` and `assetPrefix` in the Next config so that the router and asset linker know they are operating inside a sub-directory.

### **Task 3: "Ultra-Strict" TypeScript Configuration**
Standard TypeScript is safe; ultra-strict TypeScript is bulletproof. We will modify `tsconfig.json` to enforce constraints that would make an enterprise tech lead weep with joy. This proves you write flawless, deeply predictable code.

*   **Sub-task 3.1: Enable Strict Null Checks & Implicit Types.** We will ensure `strict: true` is on, but we will explicitly enforce `noImplicitAny` and `strictNullChecks`. You will be forced to handle every potential `undefined` or `null` value in your UI.
*   **Sub-task 3.2: Enable `noUncheckedIndexedAccess`.** This is a massive flex. By default, TS assumes an array `const arr = [1, 2]` has a value at `arr[5]`. Enabling this rule forces TypeScript to treat array/object access as potentially `undefined`, forcing you to write safer array iterations.
*   **Sub-task 3.3: Unused Code Prevention.** We will enable `noUnusedLocals` and `noUnusedParameters`. If you declare a variable or import a module and don't use it, the compiler will fail. The codebase remains pristine.

### **Task 4: Implementing Biome (The Modern Linter & Formatter)**
ESLint and Prettier are the tools of the past—they are slow and often conflict with each other. We will implement **Biome**, the Rust-based toolchain that formats and lints in milliseconds. This is a massive signal that you are working with the latest 2026 workflows.

*   **Sub-task 4.1: Eradicate Legacy Tooling.** We will actively remove ESLint from the `package.json` and delete the `.eslintrc.json` generated by Next.js. 
*   **Sub-task 4.2: Install and Initialize Biome.** We will install `@biomejs/biome` and generate the `biome.json` configuration file.
*   **Sub-task 4.3: Configure the Formatting Rules.** We will establish the aesthetic rules of your codebase: indent size, single vs. double quotes, trailing commas, and line width. 
*   **Sub-task 4.4: Configure the Linting Engine.** We will enable Biome's recommended ruleset, specifically turning on rules that catch React Hooks anti-patterns and unused imports.
*   **Sub-task 4.5: Establish NPM Scripts.** We will add `"check": "biome check ./src"`, `"format": "biome format --write ./src"`, and `"lint": "biome lint ./src"` to your `package.json` for easy CLI execution.

### **Task 5: The Bulletproof Git Workflow (Husky & Lint-Staged)**
We must guarantee that the virtuoso code standards established in Tasks 3 and 4 are actually enforced. We will create a Git workflow that makes it physically impossible to commit broken, unformatted, or poorly typed code.

*   **Sub-task 5.1: Install Husky.** We will install Husky and initialize it to hijack the Git hooks in your local `.git` directory.
*   **Sub-task 5.2: Install and Configure Lint-Staged.** We will install `lint-staged` and create a configuration file (`.lintstagedrc.mjs`). We will tell it to target only files currently staged for a commit (`git add`), making the pre-commit process incredibly fast.
*   **Sub-task 5.3: Construct the Pre-Commit Pipeline.** We will write the rules for lint-staged: 
    1. Run Biome to auto-format the staged files.
    2. Run Biome to lint the staged files.
    3. Run `tsc --noEmit` to compile the TypeScript abstract syntax tree and ensure no type errors exist.
*   **Sub-task 5.4: Bind to the Git Hook.** We will create the actual `pre-commit` hook file inside the Husky directory that executes the lint-staged pipeline whenever you type `git commit`. If Biome or TypeScript fails, the commit is aborted.

### **Task 6: Scalable Directory Architecture & Path Aliases**
Finally, we will set up the folder structure. A messy `src` folder is the mark of an amateur. We will create a highly organized, domain-driven structure that scales.

*   **Sub-task 6.1: Component Architecture.** We will create a `/src/components` folder, subdivided into `/ui` (for your Radix/Tailwind design system primitives) and `/layout` (for navbars and footers).
*   **Sub-task 6.2: Utilities & Hooks.** We will create `/src/lib` (for utility functions like your Tailwind class merger) and `/src/hooks` (for custom React hooks like your terminal typing simulator).
*   **Sub-task 6.3: Data & Types.** We will create a `/src/types` directory for global TypeScript interfaces and a `/src/store` directory for your Zustand state management configurations.
*   **Sub-task 6.4: Path Alias Verification.** We will verify that the `@/*` path alias in `tsconfig.json` correctly maps to this new structure, allowing us to write clean imports like `import { Button } from "@/components/ui/button"` instead of `import { Button } from "../../../components/ui/button"`.
